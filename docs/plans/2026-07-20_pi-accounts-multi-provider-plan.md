## Goal

Create the successor package `@narumitw/pi-accounts` by copying the current `pi-codex-accounts` implementation into `extensions/pi-accounts`, then archive the predecessor implementation under `deprecated/pi-codex-accounts`. Generalize only the new production package to manage OpenAI Codex (`openai-codex`), Anthropic Claude Pro/Max (`anthropic`), and GitHub Copilot (`github-copilot`). Success means each provider can independently select a named account or Pi's built-in login, refresh and activate provider-owned OAuth credentials, fail closed without cross-account fallback, and pass automated plus credential-backed runtime verification.

## Context

Pi 0.80.10 exposes provider-owned `login`, `refresh`, and `toAuth` behavior for all three target providers. OpenAI Codex and Anthropic currently derive an API key from OAuth credentials. GitHub Copilot additionally derives a credential-specific `baseUrl` and stores `enterpriseUrl` plus `availableModelIds`, so a generic implementation cannot assume that an OAuth account is only an access-token string.

The existing package already supplies private credential storage, cross-process locking, refresh serialization, race-safe runtime API-key changes, active-auth verification, Codex WebSocket invalidation, fail-closed turn abortion, and legacy credential migration. Its current implementation becomes the deprecated reference package rather than being rewritten in place; only the archived README/package metadata should gain the repository's standard deprecation notice and new directory path. The copied `pi-accounts` production package should preserve those guarantees while decomposing the 967-line Codex-specific entry point along provider, storage, runtime-auth, command, and lifecycle boundaries.

## Architecture

- Keep one package-owned `pi-accounts.json` credential store with a versioned provider map. Each provider owns an independent `active` name and an own-property account dictionary. Persist the complete JSON-safe OAuth credential returned by Pi so provider-specific fields survive refreshes and restarts.
- Define a subscription-provider adapter around Pi's built-in provider ID, display metadata, provider-owned OAuth contract, auth-application requirements, optional connection invalidator, and optional post-login/default-model behavior. Unsupported or changed capabilities reject activation before state is reported as active.
- Use a per-provider runtime-auth coordinator. Apply the `apiKey` returned by `toAuth`; temporarily overlay only the returned `baseUrl`/headers and any API-key resolver needed by an OAuth-only native provider; verify the effective key, headers, and model endpoint before success; restore the exact previous provider registration on reset or shutdown.
- Keep active selections independent. A failing selected account installs fail-closed auth and aborts turns only when the selected model uses that provider; it must not disturb active accounts for other providers.
- Expose one `/account` command with subcommands and argument completion. Mutating commands require an explicit provider ID; `default` means Pi's built-in login for that provider. Retain `/codex-login`, `/codex-account`, and `/codex-logout` as documented temporary aliases.
- Establish the successor boundary first: copy `extensions/pi-codex-accounts` to `extensions/pi-accounts`, move the predecessor to `deprecated/pi-codex-accounts`, and immediately give the new production workspace its `@narumitw/pi-accounts` identity. Generalize only the successor; add GitHub Copilot after Codex and Anthropic prove its adapter contract.

## Non-Goals

- Static API-key profile storage or switching; Pi already supports API keys through `auth.json`, environment variables, and `!command` secret-manager resolution.
- Automatic account rotation, quota evasion, or usage/quota reporting.
- Supporting xAI, Radius, custom providers, or arbitrary imported OAuth implementations in the first release.
- Publishing, running `npm deprecate` against the old npm package, or changing npm visibility without a separate release decision and explicit approval. Archiving the predecessor source under repository `deprecated/` is explicitly in scope.

## Assumptions

- The implementation targets the repository's current/latest Pi version rather than preserving the older 0.79/0.80 compatibility matrix.
- The command grammar is `/account list [provider]`, `/account login <provider> <name>`, `/account switch <provider> [name]`, and `/account remove <provider> <name>`; provider inference is not used for mutations because explicit scope is safer.
- Anthropic support means Pi's documented Claude Pro/Max OAuth flow. Documentation will state that third-party harness usage currently draws from Anthropic extra usage and is billed per token rather than consuming normal plan limits.
- The predecessor source moves to `deprecated/pi-codex-accounts` and is excluded from root workspaces, active tests, automated version bumps, and publish workflows. Its existing npm release remains installable unless a later release decision changes registry state.
- Users must not run `pi-codex-accounts` and `pi-accounts` together. Migration and installation guidance will detect or clearly reject that unsupported combination.

## Risks

- Provider registration overlays can accidentally replace models, endpoints, headers, or another extension's configuration. Snapshot/restore tests and generation ownership must make the overlay reversible and must reject activation if exact preservation is impossible.
- OAuth refresh tokens may rotate. Concurrent old/new package use or rollback to a copied legacy file can invalidate credentials, so coexistence detection and migration recovery must be designed before migration mutates files.
- GitHub Copilot account-specific model availability is credential metadata rather than plain request auth. Showing unavailable models is acceptable only as an explicitly tested/documented limitation; sending a selected account to the wrong endpoint is not.
- A generic abstraction can weaken Codex's current fail-closed and race guarantees. Shared contract tests must be added before replacing Codex-specific code, and Codex behavior must remain green throughout extraction.
- Live OAuth and model-request smokes depend on provider services and account entitlements. Deterministic automated tests remain authoritative for failure paths; unavailable external smokes must remain unchecked rather than being silently skipped.

## Rollback / Recovery

- Do not publish either npm package, run `npm deprecate`, or change registry visibility as part of implementation-only work. Moving predecessor source under repository `deprecated/` does not mutate npm registry state.
- Keep the released `@narumitw/pi-codex-accounts` npm version and its archived source available. Before the successor release, document how to remove `pi-accounts`, restore or reauthenticate the Codex credential file according to the tested migration policy, and reinstall the old package.
- Migration writes the new canonical store only under a lock and with private permissions. A failed write or validation leaves the previously usable credential source untouched; canonical state is not reported ready until it can be read back and validated.
- Runtime shutdown and rollback remove only overlays owned by the current operation generation and restore the exact prior provider configuration, leaving Pi's built-in credentials untouched.
